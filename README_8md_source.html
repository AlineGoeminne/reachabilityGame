Reachability Game
Calcule l'équilibre de Nash le plus pertinent pour un jeu d'atteignabilité quantitatif
README.md
Aller à la documentation de ce fichier.
1 # Présentation
2 Ce projet vise à avoir une idée des graphes intéressants pour la recherche d'équilibres de Nash dans des jeux d'atteignabilité sur graphe.
3 
4  - Algorithmes implémentés :
5  - DijkstraMinMax
6  - Exploration avec A*
7  - Générateurs aléatoires implémentés :
8  - Un générateur "naïf" qui génère des graphes sans contraintes sur la forme;
9  - Un générateur "arbre" qui génère au départ un arbre et qui le modifie selon les paramètres et les contraintes des jeux d'atteignabilité;
10  - Un générateur de graphes fortement connexes.
11  - Limites :
12  - Les poids sur les arcs doivent être positifs.
13  - L'exploration peut consommer beaucoup de mémoire.
14  - Il n'y a pas d'exécutable pour les tests de performance sur le générateur de graphes fortement connexes.
15  - Vertex::Ptr permet toujours de modifier le sommet même dans des cas où la modification devrait être empêchée (manque de const-correctness). Il faut donc éviter d'employer les fonctions qui modifient les sommets !
16  - Améliorations possibles :
17  - Réussir à diminuer la consommation de mémoire.
18  - Eviter de stocker les poids dans les successeurs et les prédecesseurs.
19  - Réduire la consommation de mémoire de l'exploration, si possible.
20  - Trouver une meilleure heuristique, si possible.
21  - Permettre d'avoir des poids négatifs.
22  - Permettre de charger un jeu à partir d'un fichier afin d'éviter d'avoir à recompiler à chaque nouveau graphe.
23 
24 # Compiler et exécuter
25 Il faut CMake et un compilateur C++ qui supporte le standard `C++17`. Selon le compilateur, `make` peut être nécessaire. Une fois dans la racine du projet :
26  1. Créer un dossier `build`
27  2. Rentrer dans le dossier `build`
28  3. Lancer CMake
29  - En ligne de commandes : `cmake ..`
30  - Via CMake-gui et remplissez les champs :
31  - "Where is the source code" : le chemin vers la racine du projet
32  - "Where to build the binaries" : le chemin vers le fichier `build`
33  - Via votre IDE favori
34  4. Configurer CMake
35  - Pour constuire les tests unitaires, activez `BUILD_TESTS`
36  - En ligne de commandes : `cmake -DBUILD_TESTS=TRUE ..`
37  - En cochant la case dans l'interface graphique
38  - Via votre IDE favori
39  - Pour construire les tests de performance, activez `BUILD_PERFORMANCE`
40  - En ligne de commandes : `cmake -DBUILD_PERFORMANCE=TRUE ..`
41  - En cochant la case dans l'interface graphique
42  - Via votre IDE favori
43  - Pour construire la documentation, activez `BUILD_DOC`
44  - En ligne de commandes : `cmake -DBUILD_DOC=TRUE ..`
45  - En cochant la case dans l'interface graphique
46  - Via votre IDE favori
47  5. Compiler le code
48  - En ligne de commandes (sous Linux) : `make`
49  - Via votre IDE favori
50  6. Pour générer la documentation : `make documentation`
51 
52 Les programmes seront générés dans le dossier `build` et les sous-dossiers `tests` et `performance` tandis que la documentation sera générée dans le dossier `documentation` à la racine du projet.
53 
54 Pour exécuter les tests unitaires, il suffit de lancer `ReachabilityGame-tests` dans le dossier `tests`. Pour exécuter les tests de performance, il suffit de lancer, dans le dossier `performance`, `ReachabilityGame-perf-naive` pour le générateur naïf et `ReachabilityGame-perf-tree` pour le générateur d'arbre.
55 
56 ## Résultats de performance
57 Il faut modifier les fichiers `naiveTests.cpp` et `treeTests.cpp` du dossier `performance` pour changer les paramètres à tester. Une limite de 10 secondes par jeu est fixé.
58 
59 La sortie produite (dans un fichier) devrait contenir le paramètre testé et les moyennes et médiannes des temps CPU obtenus ainsi que le nombre de fois que les 10 secondes ont été atteintes.
